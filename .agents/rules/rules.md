## 1. Project Overview
This project is a puzzle tile match game where tiles are stacked in multiple layers 
* **Responsiveness:** The game never makes the player wait while selecting tiles[cite: 1]. Logic calculations happen instantly, while DOTween + UniTask handle visual animations asynchronously.
* **Fail State:** The rack has a strict 6-slot limit. If it becomes full, the fail state triggers immediately[cite: 1].

---

## 2. Data Models (The Logic Layer)

### `LevelData` (ScriptableObject)
Generated by the Custom Editor Level Design Tool[cite: 1].
* `List<TileSaveData> activeTiles`: The baked list of starting tiles.
* `List<OrderData> pendingOrders`: The sequential list of orders to fulfill.

### `TileSaveData` (Struct/Class)
* `int tileID`: Unique logical identifier.
* `int typeID`: The icon/type identifier.
* `Vector3 visualPosition`: Stored exclusively for View instantiation.
* `List<int> blockingTileIDs`: Baked dependency list. If Count == 0, the tile is on top (clickable).

### `OrderData` (Struct/Class)
* `List<int> requiredTypeIDs`: The required sequence of tile types.
* `int currentItemIndex`: Runtime tracker for the next required icon.

### `RuntimeGameState` (Class)
* `int[] rack`: Fixed size of 6[cite: 1].
* `List<OrderData> activeOrders`: Capped at 2 simultaneous active orders.
* `Queue<OrderData> pendingOrders`: Remaining level orders.
* `Dictionary<int, TileSaveData> runtimeTiles`: Fast lookup for remaining board tiles.

## 3. System Architecture (Strict MVC)

### `Installer/` (The Composition Root)
* **`GameplayInstaller` (MonoBehaviour):** The single entry point. Initializes all pure C# Controllers in code (via constructors), injects dependencies, and wires all Signal subscriptions. Never uses the Inspector to wire callbacks or references.

### `Controller/` (Pure C# Logic - No Unity Visuals)
* **`LevelController`:** Owns `runtimeTiles`. Listens to `TileTapIntentSignal`. If `blockingTileIDs.Count == 0`, logically removes the tile from the board dictionary and passes the raw intent to `OrderController`.
* **`OrderController`:** Owns `activeOrders`. Evaluates if the incoming tile matches the `currentItemIndex` of any active order. If yes, increments progress and fires `TileRoutedToTraySignal`. If no, delegates the tile to `RackController`.
* **`RackController`:** Owns the `rack` array. Allocates non-matching tiles. If rack occupancy reaches 6, fires `RackFullSignal` (triggering Game Over immediately).
* **`GameplayController`:** Manages the high-level flow (Init, Playing, Win, Fail). Listens for end-state signals.

### `View/` (Unity MonoBehaviours - Pure Display)
* **`InputView`:** Thin input handler. Captures raycasts/taps and instantly fires `TileTapIntentSignal`. Contains zero logic and *never* injects or calls a Controller directly.
* **`VisualView`:** Owns the GameObject Pool. Listens to routing signals (not Controllers). Handles physical visual state.
  * *Performance Rule:* Uses DOTween + UniTask for async animations, strictly appending `.SuppressCancellationThrow()` to every await chain.
  * *Game Feel Rule:* Must trigger native mobile haptics (`Vibration.VibrateIOS(ImpactFeedbackStyle)`) perfectly synced with DOTween animations (e.g., Light on tap, Medium on tray route, Error on rack full).

## 4. Signal Definitions (`Signal/`)
Replaces direct C# Action events to ensure cross-layer decoupling. Passed via a lightweight event bus or signal router configured in the Installer.

* `public struct TileTapIntentSignal { public int TileID; }` 
  *(Dispatched by InputView -> Received by LevelController)*
* `public struct TileRoutedToTraySignal { public TileData Tile; public int TargetTrayIndex; }` 
  *(Dispatched by OrderController -> Received by VisualView)*
* `public struct TileRoutedToRackSignal { public TileData Tile; public int TargetRackIndex; }` 
  *(Dispatched by RackController -> Received by VisualView)*
* `public struct RackFullSignal {}` 
  *(Dispatched by RackController -> Received by GameplayController & VisualView for Game Over)*
* `public struct LevelCompletedSignal {}` 
  *(Dispatched by OrderController when pendingOrders is empty -> Received by GameplayController & VisualView for Win)*

## 5. Project Folder Architecture
Files generated by the tool must be placed strictly within these subdirectories of `Assets/_Project/`:

Assets/
├── ThirdParty/                 (Plugins: DOTween, UniTask, etc.)
└── _Project/
    ├── Controller/             (Pure C# logic: LevelController, OrderController)
    ├── Editor/                 (Level design tool scripts - MUST be in 'Editor')
    ├── Installer/              (Composition Root: GameplayInstaller)
    ├── Model/                  (Pure data: LevelData, TileData, OrderData)
    ├── Service/                (Cross-cutting: HapticService)
    ├── Signal/                 (Event bus structs: TileTapIntentSignal)
    ├── View/                   (MonoBehaviours: InputView, VisualView)
    ├── Prefabs/                (Tile visual prefabs, UI Canvas components)
    ├── Scenes/                 (Gameplay.unity)
    └── Art/                    (Sprites, UI elements)

## 6. Project Scene Hierarchy Layout
The `Gameplay.unity` scene hierarchy must be grouped under these exact organizational empty headers:

*   **`=== INSTALLER ===`** -> Holds the `[GameplayInstaller]` GameObject.
*   **`=== VIEW ===`** -> Holds `[InputView]`, `[VisualView]`, world cameras, and the core Canvas.
*   **`=== POOLS ===`** -> Sub-parent under View holding inactive runtime GameObject arrays.
